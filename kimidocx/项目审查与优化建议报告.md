## 二、🔴 严重问题（必须优先修复）

### 1. 安全漏洞：生产环境配置仍为硬编码

**后端 `back/backend/backend/settings.py`**：
- `SECRET_KEY` 仍为硬编码随机字符串
- `DEBUG = True`
- `CORS_ALLOW_ALL_ORIGINS = True`
- MySQL 密码 `YYXyyx204821` 硬编码
- `ALLOWED_HOSTS = []`

**注意**：虽然代码中已导入 `python-dotenv` 并执行 `load_dotenv()`，且目录下存在 `.env` 文件，但 **settings.py 中的实际值仍然是硬编码字符串**，环境变量并未被使用。这是一种"半修复"状态——工具到位，但未生效。

**建议**：
```python
SECRET_KEY = os.environ.get('DJANGO_SECRET_KEY')
DEBUG = os.environ.get('DJANGO_DEBUG', 'False').lower() == 'true'
CORS_ALLOW_ALL_ORIGINS = False
CORS_ALLOWED_ORIGINS = os.environ.get('CORS_ORIGINS', '').split(',')
ALLOWED_HOSTS = os.environ.get('ALLOWED_HOSTS', '').split(',')
DATABASES = {
    'default': {
        'PASSWORD': os.environ.get('DB_PASSWORD'),
        # ...
    }
}
```

**关联问题**：`data/scripts/import_to_mysql.py` 第 30 行仍硬编码了相同的数据库密码。

### 2. 模型外键映射与表结构同步风险

**架构背景**：所有业务模型均设置 `managed = False`，这是预期设计——数据库表由外部 SQL 脚本创建和维护，Django 仅作为 ORM 映射层进行查询和操作。因此"无迁移文件"对大多数业务应用来说是正常状态。

**但仍需关注的问题**：
- **外键关系**：部分模型使用 `IntegerField` 存储关联 ID 而非 `ForeignKey`，导致无法使用 `select_related` / `prefetch_related` 做关联查询优化，代码中被迫写大量原始 SQL
- **表结构同步**：如果数据库表结构发生变更（如新增字段、改枚举值），模型定义必须**手动同步更新**，否则会出现查询报错或数据映射错误
- **`warning_system.0002_add_counselor_fields`**：该迁移文件存在，需确认其是否在数据库中已应用（为 `WarningRecord` 添加了 `ai_generated_at`、`counselor_comment` 等字段）

**建议**：
- 对需要关联查询的模型，将 `IntegerField` 改为 `ForeignKey(..., db_constraint=False)`，享受 ORM 能力而不影响现有表结构
- 建立"SQL 脚本变更 → 模型同步更新"的维护流程
- 运行 `python manage.py migrate` 确认 `warning_system.0002` 状态

### 3. 后端认证配置不一致

`settings.py` 中 DRF 的认证类仅配置了：
```python
DEFAULT_AUTHENTICATION_CLASSES = [
    'rest_framework.authentication.SessionAuthentication',
    'rest_framework.authentication.BasicAuthentication',
]
```

**问题**：
- 前端 `request.js` 发送 `Authorization: Bearer <token>` 头，但后端没有配置 `JWTAuthentication`
- 这意味着前端虽然实现了 JWT 样式的 token 发送，后端实际只认 Session Cookie
- 如果部署到多域名或移动端，Session 认证会失效，而 JWT 又未真正启用

**建议**：
- 明确认证方案：要么全面启用 JWT（推荐），要么移除前端的 Bearer token 逻辑，统一使用 Session
- 如需 JWT，安装 `djangorestframework-simplejwt` 并配置

## 三、🟡 架构与工程化问题（中期优化）

### 4. 后端视图风格仍然混乱

当前后端混用至少 4 种视图编写方式：

| 应用 | 风格 | 示例 |
|---|---|---|
| `users`, `courses` | `@api_view` 装饰器函数 | `@api_view(['GET'])` |
| `learning` | `APIView` + `generics.ListAPIView` | `class LearningActivityRecordView(APIView)` |
| `algorithm`, `import_app`, `teacher_dashboard` | `APIView` 子类 | `class TrainModelView(APIView)` |
| `warning_system`, `interventions` | DRF `ModelViewSet` | `class WarningRecordViewSet(ModelViewSet)` |

**建议**：新接口统一使用 `ModelViewSet` + `Serializer` + `Router`，旧接口逐步重构。

### 7. `requirements.txt` 臃肿

文件已存在，但它是**整个 conda 环境的 `pip freeze` 导出**，包含大量与项目无关的包：
- `anaconda-navigator`、`spyder`、`jupyter`
- `torch`、`torchvision`、`torchaudio`
- `scikit-image`、`plotly`、`bokeh`

**后果**：新环境部署时会安装数百个无用依赖，极易引发版本冲突。

**建议**：
```bash
pip install pipreqs
pipreqs . --force
# 或手动清理，只保留项目实际导入的包
```
