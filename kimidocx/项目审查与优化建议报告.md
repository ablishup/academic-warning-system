# 学业预警系统 — 项目审查与优化建议报告

> 审查范围：前端（Vue 3）、后端（Django）、数据库（MySQL/SQLite）
> 审查日期：2026-04-27

---

## 一、总体评价

这是一个功能相对完整的毕业设计项目，前端 UI 较为精致，后端业务模块覆盖较全（用户、课程、学习行为、预警、干预、AI 评语、算法模型）。

但目前项目存在一个核心问题：**前后端和数据库之间存在严重的"断层"**。具体表现为：
- 数据库有两套不兼容的 SQL 方案
- Django 模型大量使用 `managed=False`，与迁移文件脱节
- 前端大量页面仍使用 mock 数据或未对接真实 API
- 9 个后端应用是空壳，处于"占位"状态

如果不先解决这些底层一致性问题，后续的功能迭代会非常困难。

## 二、🔴 严重问题（必须优先修复）

### 1. 安全漏洞：生产环境配置暴露在代码中

**后端 `back/backend/backend/settings.py`**：
- `SECRET_KEY` 使用了 Django 生成项目时的默认随机密钥（极不安全）
- `DEBUG = True`（会暴露堆栈跟踪和敏感路径）
- `CORS_ALLOW_ALL_ORIGINS = True`（任何网站都可以调用你的 API）
- MySQL 数据库密码 `YYXyyx204821` 以明文硬编码
- `data/scripts/` 下的多个脚本也硬编码了相同密码

**建议**：
```python
# 使用 python-dotenv 读取环境变量
from dotenv import load_dotenv
load_dotenv()

SECRET_KEY = os.environ.get('DJANGO_SECRET_KEY')
DEBUG = os.environ.get('DJANGO_DEBUG', 'False').lower() == 'true'
CORS_ALLOWED_ORIGINS = os.environ.get('CORS_ORIGINS', '').split(',')
DATABASES = {
    'default': {
        'PASSWORD': os.environ.get('DB_PASSWORD'),
        # ...
    }
}
```
创建 `.env` 文件并加入 `.gitignore`。

### 2. 数据库方案分裂：两套不兼容的 SQL

项目中同时存在 `data/schema/init_mysql.sql` 和 `backend/sql/chaoxing_schema.sql`，但它们的字段定义、数据类型、枚举值**完全不同**：

| 差异点 | init_mysql.sql | chaoxing_schema.sql |
|--------|---------------|---------------------|
| 学生 ID | `student_no VARCHAR(20)` | `student_id VARCHAR(50)` |
| 性别 | `TINYINT (0/1/2)` | `ENUM('男','女','未知')` |
| 预警状态 | `active/resolved/ignored` | `active/processed/resolved` |
| 课程章节 | `chapters JSON` | `total_chapters INT` |
| 作业提交 | ❌ 无 `course_id` | ✅ 有 `course_id` |

**后果**：
- `calculate_warnings.py` 查询 `courses.total_chapters` 和 `homework_submissions.course_id`，如果使用 `init_mysql.sql` 会直接报错
- `import_to_mysql.py` 的性别映射与 `chaoxing_schema.sql` 不兼容

**建议**：
1. **选定一个权威方案**（推荐 `init_mysql.sql`，因为它表更完整）
2. 删除或重命名废弃的 `chaoxing_schema.sql` 为 `chaoxing_schema.sql.bak`
3. 统一所有脚本、Django 模型、导入逻辑以适配选定的方案

---

### 3. 数据库外键缺失 + managed=False 混乱

Django 中 `Class`、`Student`、`Course` 等核心模型都设置了 `managed = False`，且关系字段全部是 `IntegerField` 而非 `ForeignKey`：

```python
# 当前（错误）
class Student(models.Model):
    class_id = models.IntegerField()  # 没有外键约束
    major_id = models.IntegerField()
```

**后果**：
- 数据库层面没有引用完整性保护，可以插入不存在的 class_id
- Django ORM 无法使用 `select_related`、`prefetch_related` 做关联查询优化
- 代码中被迫写大量原始 SQL（如 `SELECT id FROM users WHERE username = %s`）

**建议**：
如果表确实由 SQL 脚本创建，Django 只负责查询，也应该在模型中使用 `ForeignKey` + `db_column` + `db_constraint=False`：

```python
class Student(models.Model):
    class_info = models.ForeignKey(
        Class, on_delete=models.DO_NOTHING,
        db_column='class_id',
        db_constraint=False  # 不创建 DB 级外键约束
    )
```
这样既能享受 ORM 的关联查询能力，又不会影响现有表结构。

### 4. 干预模块重复：`interventions` vs `intervention_system`

两个应用实现了同一套业务，但表结构完全不同：

| | interventions | intervention_system |
|---|---|---|
| 表名 | `intervention_records` | `interventions_record` |
| 效果字段 | `is_effective` (SmallInteger) | `effectiveness` (CharField) |
| 随访 | `follow_up_needed` (SmallInteger) | `InterventionFollowUp` 独立表 |

**后果**：URL 路由只注册了 `interventions`，`intervention_system` 的所有接口都无法访问。如果有人在 `intervention_system` 上开发，数据会写入完全不同的表。

**建议**：
- 对比两个模型，保留字段更完整、设计更合理的一个
- 将另一个应用的数据迁移到主表后删除
- 统一前端调用的 API 路径

---

### 5. 前端：零路由守卫 + Pinia 完全未使用

**路由守卫缺失**：`router/index.js` 中没有 `beforeEach`，任何用户直接在地址栏输入 `/admin`、`/teacher`、`/counselor` 都能进入对应页面，仅靠后端 API 拒绝请求来防护。

**Pinia 空转**：`main.js` 中 `app.use(createPinia())`，但 `src/stores/` 目录下没有任何 store 文件。所有状态靠 `localStorage` + 组件局部 `ref` 管理，导致：
- `TeacherLayout` 和 `CounselorLayout` 各自独立请求用户信息
- 没有全局加载状态、没有登录态统一判断
- 刷新页面后各布局重复发请求

**建议**：
1. 创建 `src/stores/auth.js` 管理 token、用户信息、角色
2. 创建 `src/stores/user.js` 缓存当前用户资料
3. 在 `router/index.js` 添加导航守卫：

```js
router.beforeEach((to, from, next) => {
  const token = localStorage.getItem('token')
  if (to.path !== '/login' && !token) {
    return next('/login')
  }
  // 角色权限校验...
  next()
})
```

### 6. 前端 API 导出不完整

`src/api/index.js` 只导出了 `student` 和 `teacher`，`admin` 和 `counselor` 模块被遗漏：

```js
export * from './student'
export * from './teacher'
// admin 和 counselor 缺失！
```

这会导致视图组件中如果通过 `import { xxx } from '@/api'` 引入 admin/counselor 接口会报错。

### 7. 迁移未应用：`warning_system.0002`

`python manage.py showmigrations` 显示 `warning_system` 的 `0002_add_counselor_fields` 处于 `[ ]` 未应用状态。该迁移为 `WarningRecord` 新增了 `ai_generated_at`、`counselor_comment` 等字段。如果生产环境的数据库没有这些字段，相关功能会直接报错。

**建议**：
```bash
cd back/backend
python manage.py migrate warning_system
```

---

### 8. 数据脚本中的静默错误吞食

`import_to_mysql.py` 和 `calculate_warnings.py` 中存在大量：

```python
try:
    cursor.execute(...)
except:
    pass
```

这意味着任何 SQL 错误（语法错误、外键冲突、数据类型不匹配）都会被无声忽略，你完全不知道导入失败了多少条数据。

**建议**：至少改为：

```python
except Exception as e:
    print(f"导入失败: {e}, 数据: {row}")
    # 或写入日志文件
```

## 三、🟡 架构与工程化问题（中期优化）

### 9. 后端：空壳应用过多

以下 9 个应用是**完全空的**（只有默认的 `# Create your models here.`）：

`analysis`, `attendances`, `auth`, `resources`, `scores`, `student_analysis`, `school_analysis`, `student_interventions`, `student_warnings`

**建议**：
- 如果确定不再需要，直接删除这些应用目录（同时清理 `INSTALLED_APPS` 和根 `urls.py`）
- 如果计划后续开发，保留但添加 TODO 注释说明规划用途，避免审查时被认为是未完成

---

### 10. 后端：视图风格混乱

同一个项目中混用了至少 4 种视图编写方式：
- `@api_view` 装饰器函数视图（users, courses）
- DRF 通用类视图（learning, warning_system）
- `APIView` 子类（algorithm, teacher_dashboard）
- Django 原生视图（import_app）

**建议**：统一使用 DRF 的 `ModelViewSet` + `Serializer` 模式，代码量可以减少 30%-50%，且自动获得列表、详情、创建、更新、删除接口。

---

### 11. 后端：多处重复代码未抽象

`get_student_from_user()` 这个辅助函数在 `courses/views.py`、`warning_system/views.py`、`learning/views.py` 中**几乎一模一样**地复制了多次。

**建议**：提取到公共模块，例如 `back/backend/common/utils.py` 或 `users/utils.py`。

---

### 12. 前端：视图文件过大

| 文件 | 行数 |
|---|---|
| `counselor/Interventions.vue` | 1599 行 |
| `counselor/Warnings.vue` | 1223 行 |
| `student/Analysis.vue` | 896 行 |

这些文件承担了列表展示、弹窗表单、详情页、图表渲染等多重职责。

**建议**：按职责拆分为子组件：
```
views/counselor/Warnings/
  ├── index.vue          # 主页面（列表+搜索）
  ├── WarningDetail.vue  # 详情弹窗
  ├── SendSMSDialog.vue  # 发送短信弹窗
  └── StudentGroupPanel.vue # 学生分组面板
```

---

### 13. 前端：四个 Layout CSS 大量重复

`Layout.vue`、`TeacherLayout.vue`、`CounselorLayout.vue`、`AdminLayout.vue` 各自维护了几乎完全相同的侧边栏样式（约 150 行 CSS）。

**建议**：
- 提取一个 `BaseLayout.vue`，包含通用的侧边栏/头部/主内容区结构
- 各角色 Layout 只覆盖少量差异（如菜单项、颜色主题）
- 或使用 CSS 变量定义主题色，避免硬编码渐变值

---

### 14. 没有 `requirements.txt`

项目没有记录依赖版本，新环境部署时只能靠猜测安装包。

**建议**：
```bash
cd back/backend
pip freeze > requirements.txt
```
并清理其中非项目依赖的系统包。

## 四、🟢 优化建议（提升体验）

### 15. 前端 mock 数据清理

以下位置仍使用硬编码假数据：
- `views/admin/Dashboard.vue` — 所有统计数据、图表、表格都是静态写死的
- `api/counselor.js` 的 `getClassStats()` — 直接返回模拟数组
- `counselor/Dashboard.vue` 中的 `getMockStudentWarnings()`

**建议**：尽快对接后端真实 API，或在 `NODE_ENV === 'development'` 时才启用 mock。

---

### 16. 清理生产环境调试代码

项目中散布着 40 余处 `console.log` / `console.error`，包括：
- `console.log('API响应:', res)`
- `console.log('学生数据:', students.value)`

**建议**：全局搜索 `console.log` 并删除，或使用封装后的日志工具（开发环境打印，生产环境静默）。

---

### 17. 上传地址硬编码

`UploadData.vue` 中：
```js
const uploadUrl = 'http://localhost:8000/api/import/activities/'
```
这会导致部署到服务器后上传失败。

**建议**：使用 axios 实例的 baseURL，或从环境变量读取：`import.meta.env.VITE_API_BASE_URL`

---

### 18. `/admin/system` 有菜单无页面

`AdminLayout.vue` 中有系统设置菜单链接到 `/admin/system`，但路由表和视图文件都不存在。

**建议**：要么补充该页面，要么从菜单中移除该项。

---

### 19. `knowledge_mastery` 表无数据填充

`chaoxing_schema.sql` 中定义了 `knowledge_mastery` 表，但：
- `init_mysql.sql` 没有这张表
- `import_to_mysql.py` 中的 `calculate_knowledge_mastery()` 是空实现（注释说"将通过API动态计算"）
- `calculate_warnings.py` 查询了这张表，但没有任何地方往里写数据

**后果**：学生的"知识点掌握度"和"学习进度"分数可能始终为 0。

**建议**：
- 明确该表的数据来源（是通过学习行为日志计算，还是教师手动录入？）
- 如果是计算得出，在 `calculate_warnings.py` 中补充计算逻辑
- 如果是手动录入，提供教师端录入接口

---

### 20. 缺少全局错误边界和加载状态

前端每个页面都独立写 `try/catch + ElMessage.error`，后端很多视图用裸 `except Exception as e:`。

**建议**：
- 前端：在 `request.js` 响应拦截器中统一处理 401/403/500，配合 Element Plus 的全局 message
- 后端：使用 DRF 的异常处理中间件统一包装错误响应
